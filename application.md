**[ НАЗВАНИЕ ПРОЕКТА ]**

"Co-Study - веб-сервис для совместной учебы и проектной работы"

**[ ПРОБЛЕМНОЕ ПОЛЕ ]**

Современный учебный процесс требует от старшеклассников и студентов высокого уровня самоорганизации, особенно при подготовке к экзаменам и выполнении групповых проектов. Основными проблемами в этой сфере являются: 

1. Низкая мотивация при самостоятельной подготовке 
2. Сложности с координацией задач внутри учебных мини-команд
3. Отдельное использование мессенджеров и трекеров задач 

Существующие зарубежные платформы для совместной учебы либо перегружены тяжелым видеостримингом (что создает технические сбои), либо ориентированы на случайный мэтчинг с незнакомцами. В них **отсутствует**: 

1. Удобный фокус на создание закрытых локальных групп (например, класса или проектной команды лицея)
2. Интеграция совместного таймера с общим канбан-трекером задач
3. Удобная среда для быстрого текстового обсуждения и синхронной работы над проектами

Заявляемый веб-сервис позволит решить эти проблемы и обеспечит пользователям единый цифровой коворкинг, объединяющий управление задачами и элементы взаимной мотивации в рамках единого интерфейса с облачным сохранением данных.

**[ ЗАКАЗЧИК / ПОТЕНЦИАЛЬНАЯ АУДИТОРИЯ ]**

Продукт нацелен на учащихся, практикующих совместное обучение и проектную деятельность:
* Учащиеся лицея НИУ ВШЭ и других старших школ, готовящиеся к экзаменам и выполняющие групповые исследования
* Студенты вузов, организующие совместную подготовку к сессиям
* Самостоятельные учебные мини-команды, которым нужен единый инструмент для координации

**[ АППАРАТНЫЕ / ПРОГРАММНЫЕ ТРЕБОВАНИЯ ]** 

Продукт разрабатывается как кроссплатформенный веб-сервис (Web-first), оптимизированный для работы на различных устройствах.
* **Клиентская часть:** Любые устройства (ПК, ноутбуки, планшеты, смартфоны) с доступом в интернет
* **Программные требования:** Актуальные версии браузеров (Google Chrome, Yandex, Safari)
* **Серверная часть:** Удаленный облачный сервер для бэкенда и базы данных, обеспечивающий доступ к профилю по логину/паролю с любого устройства

**[ ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ ]**

Программный продукт будет предоставлять следующие возможности:
* Регистрация, авторизация пользователей и создание индивидуального профиля
* Создание закрытых и публичных виртуальных "учебных комнат" по уникальной ссылке или коду
* Синхронный таймер Помодоро внутри комнаты, единый для всех её участников (работает в реальном времени через веб-сокеты)
* Командная канбан-доска внутри комнаты (создание, перетаскивание, назначение и удаление карточек задач)
* Текстовый чат внутри комнаты 
* Сохранение всей истории, прогресса задач и созданных материалов в облачной базе данных (синхронизация между устройствами пользователя)

**[ ПОХОЖИЕ / АНАЛОГИЧНЫЕ ПРОДУКТЫ ]**

Анализ 3 программных продуктов, которые частично пересекаются с заданным функционалом, показал, что:
* **StudyStream:** не содержит инструментов менеджмента задач (канбан-досок) и не подходит для структурированной работы небольших проектных команд
* **Focusmate:** имеет жесткое ограничение по времени сессий
* **Trello:** не имеет встроенных учебных таймеров и чатов реального времени для живого общения

**[ ИНСТРУМЕНТЫ РАЗРАБОТКИ, ИНФОРМАЦИЯ О БД ]**

* **Frontend:** HTML5, CSS3, JavaScript (библиотека React.js)
* **Backend:** Python (фреймворк FastAPI), библиотека WebSockets для обеспечения интерактивности
* **База данных:** PostgreSQL (развернутая на удаленном облачном хостинге)

**[ ЭТАПЫ РАЗРАБОТКИ ]**

* Проектирование схемы базы данных и логики серверных запросов (API)
* Разработка бэкенда: авторизация пользователей и логика управления учебными комнатами
* Реализация облачного хранения данных (пользователи, комнаты, канбан-задачи) в PostgreSQL
* Настройка интерактивной связи (WebSockets) для мгновенного обновления таймера и чата
* Создание веб-интерфейса (React.js) с использованием готовых компонентов для ускорения разработки
* Связывание фронтенда и бэкендом в единую систему
* Развертывание бэкенда и базы данных на удаленном облачном хостинге
* Тестирование основных сценариев (вход, создание комнаты, перемещение задач) и защита проекта

**[ ВОЗМОЖНЫЕ РИСКИ ]**

* Трудности с первоначальной настройкой двусторонней связи через WebSockets для синхронизации таймера без задержек
* Ошибки при проектировании связей в базе данных (многие-ко-многим между пользователями, комнатами и задачами), что может потребовать рефакторинга кода
* Риск неправильного распределения времени на изучение выбранного стека технологий (React/FastAPI) на глубоком уровне
